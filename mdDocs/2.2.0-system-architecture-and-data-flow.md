## 2.0 Authentication Flow Sequence Diagram

```
Android Client                     Backend API                  Database
     │                                  │                            │
     │                                  │                            │
     │  1. User enters email             │                            │
     │  2. POST /auth/getOtp             │                            │
     ├─────────────────────────────────>│                            │
     │                                  │  Generate OTP              │
     │                                  │  Send to email             │
     │                                  │                            │
     │  3. User receives OTP in email    │                            │
     │                                  │                            │
     │  4. User enters OTP               │                            │
     │  5. POST /auth/getNewToken        │                            │
     ├─────────────────────────────────>│                            │
     │                                  │  Verify OTP                │
     │                                  │  Generate JWT (7 days)     │
     │                                  │  Return token              │
     │  6. Receive JWT token             │                            │
     │<─────────────────────────────────┤                            │
     │                                  │                            │
     │  7. Store token securely          │                            │
     │     (EncryptedSharedPreferences)  │                            │
     │                                  │                            │
     │  8. Get FCM token from Firebase   │                            │
     │                                  │                            │
     │  9. POST /auth/saveUser           │                            │
     │     (with JWT auth header)        │                            │
     ├─────────────────────────────────>│                            │
     │                                  │  Create User record        │
     │                                  ├───────────────────────────>│
     │                                  │  Store messagingToken      │
     │                                  │<───────────────────────────┤
     │                                  │  Return user data          │
     │  10. User registration complete   │                            │
     │<─────────────────────────────────┤                            │
     │                                  │                            │
     │  11. Token auto-included in       │                            │
     │      all subsequent requests      │                            │
     │      (Authorization: Bearer)      │                            │
     │                                  │                            │

Note: All requests after step 9 must include:
      Authorization: Bearer {jwt_token}
```

## 4.0 Carpool Joining Flow (Ride Seeker)

### 4.1 Scenario A: Matching Trip Found

```
Seeker Application                Backend API                  Database      Notifications
     │                                  │                            │                │
     │  1. User clicks "Find Carpool"   │                            │                │
     │  2. Shows search form            │                            │                │
     │     - Time selector              │                            │                │
     │     - Location selector          │                            │                │
     │                                  │                            │                │
     │  3. User selects origin          │                            │                │
     │  4. User selects destination     │                            │                │
     │                                  │                            │                │
     │  5. User clicks "Search"         │                            │                │
     │  6. POST /trips/joinCarPool      │                            │                │
     │     (with JWT auth)              │                            │                │
     ├─────────────────────────────────>│                            │                │
     │                                  │                            │                │
     │                                  │  1. Geocode destination    │                │
     │                                  │     to find zone           │                │
     │                                  │     (e.g., "Kileleshwa")   │                │
     │                                  │                            │                │
     │                                  │  2. Query database:        │                │
     │                                  │     SELECT trips WHERE     │                │
     │                                  │     - zone = "Kileleshwa"  │                │
     │                                  │     - status = OPEN        │                │
     │                                  │     - departureTime ≈      │                │
     │                                  │       user's time          │                │
     │                                  ├───────────────────────────>│                │
     │                                  │  Found: Trip #123          │                │
     │                                  │<───────────────────────────┤                │
     │                                  │                            │                │
     │                                  │  3. Add user to trip       │                │
     │                                  │     UPDATE trip SET        │                │
     │                                  │     capacity--,            │                │
     │                                  │     passengers+=1          │                │
     │                                  ├───────────────────────────>│                │
     │                                  │<───────────────────────────┤                │
     │                                  │                            │                │
     │                                  │  4. If capacity = 0:       │                │
     │                                  │     UPDATE trip SET        │                │
     │                                  │     status = FULL          │                │
     │                                  │                            │                │
     │                                  │  5. Get trip details       │                │
     │                                  │  6. Get host's user info   │                │
     │                                  │     (name, rating, etc.)   │                │
     │                                  │     [Future: ratings]      │                │
     │                                  │                            │                │
     │                                  │  7. Send FCM notification  │                │
     │                                  │     to HOST                │────────────────>│
     │                                  │     "John joined your trip"│  Firebase      │
     │                                  │<───────────────────────────────────────────│
     │                                  │                            │                │
     │                                  │  8. Return trip details    │                │
     │  7. Receive success (HTTP 200)   │                            │                │
     │<─────────────────────────────────┤                            │                │
     │     - tripId                     │                            │                │
     │     - hostName                   │                            │                │
     │     - capacity (remaining seats) │                            │                │
     │     - polyline                   │                            │                │
     │     - departure time             │                            │                │
     │                                  │                            │                │
     │  8. Display trip confirmation    │                            │                │
     │     - "Successfully joined!"     │                            │                │
     │     - Show host details          │                            │                │
     │     - Show trip on map           │                            │                │
     │     - Countdown timer to pickup  │                            │                │
     │                                  │                            │                │
     │  9. User is now part of trip     │                            │                │
```

## 5.0 Real-Time Notification Flow

```
Backend Automation                Firebase Service              Android Device
     │                                  │                            │
     │  [Event: User joined trip]       │                            │
     │  [Event: Trip cancelled]         │                            │
     │  [Event: User matched to trip]   │                            │
     │                                  │                            │
     │  TripManagementService           │                            │
     │  FirebaseMessagingService        │                            │
     │  (orchestrates notifications)    │                            │
     │                                  │                            │
     │  1. Get recipient's FCM token    │                            │
     │     from database                │                            │
     │                                  │                            │
     │  2. Build notification message   │                            │
     │     with context                 │                            │
     │                                  │                            │
     │  3. Send to Firebase API         │                            │
     ├─────────────────────────────────>│                            │
     │                                  │  FCM routing logic         │
     │                                  │  - Check device online     │
     │                                  │  - Route to correct device │
     │                                  │                            │
     │                                  │  Send notification         │
     │                                  ├───────────────────────────>│
     │                                  │                            │
     │                                  │  SwoopMessagingService     │
     │                                  │  onMessageReceived()       │
     │                                  │                            │
     │                                  │  Display notification      │
     │                                  │  in system tray            │
     │                                  │<───────────────────────────┤
     │                                  │                            │
     │                                  │  User taps notification    │
     │                                  │  → Intent to trip details  │
     │
Notification Types Sent:
- "Carpool Matched"
- "Trip Cancelled" 
- "Passenger Joined"
- "Trip Starting Soon" [Future]
- "Emergency Alert" [Future]
```

## 6.0 Data Models & Relationships

### 6.1 User Entity

```
┌─ User ──────────────────────────┐
│ userId: UUID (PK)               │
│ fullName: String                │
│ email: String (UNIQUE)          │
│ role: RIDE_SEEKER | CARPOOL_HOST│
│ messagingToken: String          │
│ createdAt: LocalDateTime        │
│ updatedAt: LocalDateTime        │
├─ Relationships ─────────────────┤
│ → Vehicle (1:1, optional)       │
│ → Trip (M:M as passenger)       │
│ → Trip (1:M as creator)         │
│ → RideSeekerBacklogEntry (1:M)  │
└─────────────────────────────────┘
```

### 6.2 Trip Entity

```
┌─ Trip ──────────────────────────────┐
│ tripId: UUID (PK)                   │
│ tripCapacity: int (1-7)             │
│ tripStatus: OPEN | FULL | CANCELLED │
│ departureTime: LocalDateTime        │
│ destinationZone: String (from G.Maps)
│ routePolyline: String (encoded)     │
│ createdBy: UUID (FK → User)         │
│ originDestination: Embeddable       │
│   - originLatitude: Double          │
│   - originLongitude: Double         │
│   - destinationLatitude: Double     │
│   - destinationLongitude: Double    │
│ createdAt: LocalDateTime            │
│ updatedAt: LocalDateTime            │
├─ Relationships ─────────────────────┤
│ → Vehicle (optional, 1:1)           │
│ ← User[] (passengers, M:M)          │
│ ← RideSeekerBacklogEntry[] (1:M)    │
└─────────────────────────────────────┘
```

### 6.3 Vehicle Entity

```
┌─ Vehicle ──────────────────────┐
│ vehicleId: UUID (PK)           │
│ vehicleRegNumber: String       │
│ vehicleDescription: String     │
│ userId: UUID (FK → User)       │
│ createdAt: LocalDateTime       │
│ updatedAt: LocalDateTime       │
├─ Relationships ────────────────┤
│ → User (1:1)                   │
│ ← Trip[] (optional, 1:M)       │
└────────────────────────────────┘
```

### 6.4 RideSeekerBacklogEntry Entity

```
┌─ RideSeekerBacklogEntry ───────┐
│ backlogEntryId: UUID (PK)      │
│ userId: UUID (FK → User)       │
│ destinationZone: String        │
│ matched: boolean               │
│ requestMadeAt: LocalDateTime   │
│ matchedAt: LocalDateTime       │
│ createdAt: LocalDateTime       │
│ updatedAt: LocalDateTime       │
├─ Relationships ────────────────┤
│ → User (M:1)                   │
│ → Trip (optional, M:1) [Future]│
└────────────────────────────────┘
```

## 8.0 Error Handling & Retry Strategy

### 8.1 Retry Logic Decision Tree

```
API Request
    │
    ├─ Success (200-202) ──────────> Return success
    │
    ├─ Client Error (4xx)
    │  ├─ 400 (Bad Request) ──────> User fix required (no retry)
    │  ├─ 401 (Unauthorized) ─────> Refresh token → Retry (1x)
    │  ├─ 403 (Forbidden) ────────> Permission error (no retry)
    │  └─ 409 (Conflict) ────────> Already exists (no retry)
    │
    ├─ Server Error (5xx)
    │  └─ Exponential backoff + Retry
    │     Attempt 1: Wait 1s → Retry
    │     Attempt 2: Wait 2s → Retry
    │     Attempt 3: Wait 4s → Fail
    │
    ├─ Network Error (No Connection)
    │  └─ Exponential backoff + Retry
    │     (Same as 5xx)
    │
    └─ Timeout
       └─ Exponential backoff + Retry
          (Same as 5xx)
```

### 8.2 Retrofit Interceptor for Auto-Retry

```kotlin
class RetryInterceptor : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val request = chain.request()
        var response: Response? = null
        var exception: Exception? = null
        
        for (attempt in 0..2) {
            try {
                response = chain.proceed(request)
                
                // Don't retry on client errors
                if (response.code in 400..499) return response
                
                // Retry on server errors and success
                if (response.isSuccessful || response.code in 500..599) {
                    return response
                }
            } catch (e: Exception) {
                exception = e
                if (attempt < 2) {
                    Thread.sleep(1000 * (1 shl attempt)) // Exponential backoff
                }
            }
        }
        
        return response ?: throw exception ?: IOException("Unknown error")
    }
}
```

## 9.0 State Management in ViewModel

### 9.1 Recommended ViewModel State Pattern

```
ViewModel State:
  ├─ Loading State (showing progress)
  ├─ Success State (data loaded, ready to display)
  ├─ Error State (error occurred, show message)
  └─ Empty State (no data available)

Flow:
  Idle → Loading → Success/Error/Empty → Idle
```

### 9.2 ViewModel Implementation Example

```kotlin
sealed class UiState<T> {
    class Idle<T> : UiState<T>()
    class Loading<T> : UiState<T>()
    data class Success<T>(val data: T) : UiState<T>()
    data class Error<T>(val message: String) : UiState<T>()
    class Empty<T> : UiState<T>()
}

class TripViewModel(private val repository: TripRepository) : ViewModel() {
    private val _tripState = MutableLiveData<UiState<TripData>>(UiState.Idle())
    val tripState: LiveData<UiState<TripData>> = _tripState
    
    fun createTrip(tripRequest: CreateTripRequest) {
        viewModelScope.launch {
            _tripState.value = UiState.Loading()
            
            try {
                val result = repository.createTrip(tripRequest)
                if (result.isSuccess) {
                    _tripState.value = UiState.Success(result.getOrNull()!!)
                } else {
                    _tripState.value = UiState.Error(result.exceptionOrNull()?.message ?: "Unknown error")
                }
            } catch (e: Exception) {
                _tripState.value = UiState.Error(e.message ?: "Unknown error")
            }
        }
    }
}

// In Activity/Fragment
tripViewModel.tripState.observe(this) { state ->
    when (state) {
        is UiState.Idle -> {
            // Initial state
        }
        is UiState.Loading -> {
            progressBar.visibility = View.VISIBLE
            button.isEnabled = false
        }
        is UiState.Success -> {
            progressBar.visibility = View.GONE
            button.isEnabled = true
            displayTripDetails(state.data)
        }
        is UiState.Error -> {
            progressBar.visibility = View.GONE
            button.isEnabled = true
            showErrorDialog(state.message)
        }
        is UiState.Empty -> {
            progressBar.visibility = View.GONE
            showEmptyState()
        }
    }
}
```

## 10.0 Sequence Summary: Complete User Journey

```
1. AUTHENTICATION
   ├─ Request OTP
   ├─ Verify OTP → Get JWT
   ├─ Register User (with FCM token)
   └─ Store JWT securely

2. SETUP
   ├─ HOST: Register vehicle
   └─ SEEKER: Set preferences

3. CARPOOL HOST FLOW
   ├─ Click "Create Trip"
   ├─ Fill trip details (time, capacity, location)
   ├─ POST /trips/createTrip
   ├─ Backend gets route from Google Maps
   ├─ Trip created, OPEN for joiners
   ├─ Receive FCM when user joins
   ├─ Show passenger details
   └─ Cancel trip if needed → Passengers notified

4. RIDE SEEKER FLOW
   ├─ Click "Find Carpool"
   ├─ Fill search (time, location)
   ├─ POST /trips/joinCarPool
   ├─ Case A: Match found
   │  ├─ Receive success response
   │  ├─ Show trip details
   │  └─ View host information
   ├─ Case B: No match
   │  ├─ Added to backlog (HTTP 202)
   │  ├─ Wait for notification
   │  ├─ FCM "Carpool matched!"
   │  └─ Auto-joined to trip
   └─ View trip on map with polyline

5. ONGOING
   ├─ Both see trip status updates
   ├─ Receive notifications (joined, cancelled, etc.)
   ├─ In future: ratings, in-app chat
   └─ Trip completion
```
