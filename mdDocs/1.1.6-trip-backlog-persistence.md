# 1.1.6 Trip Backlog Persistence

## 1.1.6.1 Overview

The `tripManagement` backlog now persists ride-seeker requests in MySQL instead of keeping them in an in-memory queue. This ensures backlog records survive application restarts and can be audited after matching.

## 1.1.6.2 Backlog Table Purpose

The `ride_seeker_backlog` table stores unmatched ride requests for later onboarding when a compatible host creates a trip.

Each record includes:

- `backlogEntryId`
- `user`
- `destinationZone`
- `requestMadeAt`
- `matched`
- `matchedAt`
- `createdAt`
- `updatedAt`

## 1.1.6.3 Record Creation Rules

A new backlog record is created when:

1. A ride seeker tries to join a carpool and no open trip is available.
2. A host cancels an open trip and existing passengers must be re-queued.

In both cases, the request timestamp is the time the backlog row is persisted.

## 1.1.6.4 Matching Rules During Trip Creation

When `TripManagementService.createTrip(...)` creates a trip:

1. The destination zone is resolved from Google Maps.
2. The backlog is queried in ascending `requestMadeAt` order.
3. Only unmatched backlog rows with a similar destination zone are considered.
4. The oldest compatible rows are selected up to the available trip capacity.
5. Each selected backlog row is marked as matched and stamped with `matchedAt`.
6. Each matched user is added to the trip and notified.

Destination-zone similarity is normalized to tolerate formatting differences such as:

- `THIKA_ROAD`
- `Thika Road`
- `thika-road`

## 1.1.6.5 Persistence Notes

- Matching state is stored in the backlog row rather than deleting the row.
- Trip creation, cancellation, and backlog updates are executed transactionally in the service layer.
- The MySQL schema is created automatically through Hibernate because `spring.jpa.hibernate.ddl-auto=update` is enabled in the current development configuration.
